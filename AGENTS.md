## 核心准则（Karpathy 准则）

### 1. 编码前先思考
- 明确假设，不确定时直接提问
- 多种理解时列出选项，不擅自决定
- 有更简单方案时主动提出，该反驳时反驳
- 遇到不清楚的地方，停下并指出困惑点

### 2. 简洁优先
- 最少代码解决问题，不做推测性实现
- 不添加超出需求的功能
- 不为一次性代码写抽象层
- 不添加未被要求的"灵活性"或"可配置性"
- 不写不可能场景的错误处理
- 写了200行而50行能搞定，重写

### 3. 精准修改
- 只动必须动的地方，只清理自己制造的混乱
- 不"改进"相邻代码、注释或格式
- 不重构没坏的东西
- 匹配现有代码风格，即使自己会写得不一样
- 删除自己修改导致不再使用的导入/变量/函数
- 预先存在的死代码只提不删，除非被要求

### 4. 目标驱动执行
- 编码前定义可验证的成功标准
- 将任务转化为可测试目标：
  - "添加验证" → "为无效输入写测试，然后让它们通过"
  - "修复bug" → "写复现测试，然后让它通过"
  - "重构X" → "确保重构前后测试都通过"
- 多步骤任务列出计划：步骤 → 验证点

## 提交规范（Git）

### 默认策略（与用户意图对齐）

- **默认不代劳**：未收到用户明确要求时，**不执行** `git commit` / `git push`，也**不因循例反复追问**要不要提交。
- **每批改动结束交付物**：必须给出「变更文件清单 + 关键意图 + 验证方式（编译/测试/必要的点操）」，便于人肉或另一位助手接手。

### 当用户明确要求「提交 / 推送」时

- 只允许提交与用户指令一致的范围；必要时先 `git status`/`git diff` 自我核对，避免把 `.db`、`bin`、`obj`、`.vs` 等产物夹带进提交。
- **禁止擅自进行危险 Git 操作**：如对已共享分支的 **`git push --force`**、改写历史的 **`git reset --hard` / `git rebase`** 等（除非用户**逐字确认**后果、分支名与影响范围）；禁止把「整理提交」与用户未点名的改动绑在一起。
- **提交信息格式**：
  - `feat:` — 新功能
  - `fix:` — Bug 修复
  - `refactor:` — 代码重构
  - `docs:` — 文档/注释更新
  - `test:` — 测试相关
  - `chore:` — 构建、配置、清理等
- **推送前核对**：远程名（通常是 `origin`）、分支名、`git remote -v` 是否与目标仓库一致。
- **提交前检查项**：
  - [ ] 编译是否通过（如有项目文件变更）
  - [ ] 是否有敏感信息泄露（密码、密钥、绝对路径等）
  - [ ] `.vs/` / `bin/` / `obj/` 等忽略目录是否被误跟踪
  - [ ] SQLite 数据库文件未被跟踪（`*.db`、`*.sqlite*` 等应保持忽略）

## 人机协作约定（KimiCode × Cursor Agent）

为避免「两个助手轮流改同一模块」时出现风格漂移与安全回退，建议固定以下流程：

### 并行协作原则

- **按模块领任务**：尽量让单次 PR/会话只触碰一个高风险域（`Safety`、`StateMachine`、`Data` Schema、`Motion` 硬件抽象之一），降低合并冲突与隐性行为变化。
- **先拉再改**：开始改代码前应基于最新 `master`（或约定的集成分支）；若与他人/另一助手并行，优先 **小而频** 的合并，避免长寿命大分支。
- **禁止悄悄夹带**：不把无关重构、大范围格式化、顺手重命名与需求混在同一提交里；工业现场回滚成本很高。

### Safety / 运动 / 状态机 的硬门槛

- **Safety 相关改动**：默认应附 **单测** 或（若硬件相关）明确写出**点操验证步骤**与**残余风险**（例如 AABB 近似带来的漏检场景）。
- **互锁与监控路径**：`InterlockEngine` 的条件评估、`SafetyMonitor` 周期线程里，**禁止阻塞**；不要引入长时间 IO/锁等待。
- **生产发布**：严禁以运行时可开关的方式削弱安全链（例如 `SafeMotionController.SafetyEnabled = false` 出现在产线配置路径）；若确需调试隔离，必须限定在 **Debug 编译** 或独立诊断开关，并在文档中写明边界。

### 文档与事实来源

- **实现以代码为准**：若 `README` / `SAFETY.md` 与代码行为不一致，优先改文档或改代码二选一，并在提交说明里写清楚「修正文档」或「修正行为」。
- **个人计划路径**（如本机 `C:\Users\...\.kimi\plans\...`）不应被当作团队事实标准；跨机器协作请把结论写回仓库内文档或 PR 描述。

### 单次会话 / PR 建议节奏（与其它助手轮转时尤其重要）

1. **先对齐上下文**：分支名、本轮目标、刻意不验证的部分（若有）。
2. **按主轴划界**：尽量让单次改动只啃一个高风险域（`Safety` / `StateMachine` / `Data.Schema` / `Motion`）。
3. **改完自检**：至少有 **build**；能跑则用 **test**；涉及 UI 则说明需要点哪里看现象。
4. **文档同窗**：改了对外可见行为（安全策略、初始化、数据库文件名/路径），在同一批改动里同步 `README`/`SAFETY.md`/本文件。

**旧版 AGENTS.md 中与此冲突的说法**（如「永远不讨论」「永不执行 Git」「只管贴代码不给说明」）：**一律以本节、上文「输出与沟通」与「提交规范（Git）」为准**。

## 编码风格

- **C# 命名规范**：PascalCase（类/方法/属性）、camelCase（局部变量/参数）、_camelCase（私有字段）
- 字段用 `private readonly`，只在必要时 `public`
- 公共 API 必须添加 XML 文档注释
- 异步方法以 `Async` 结尾，返回 `Task` 或 `Task<T>`
- 优先使用 `var` 进行局部变量声明；**Halcon API 优先用显式类型，不用 `var`**
- 异常处理：捕获具体类型，不静默吞掉
- XAML：生产代码不加设计时 `d:` 属性

## 输出与沟通（回答长什么样）

### 默认（推荐：可审查、可交接）

- **先短后长**：用少量文字说明「改了什么 / 为什么 / 怎么验证」，再给出必要代码引用或补丁；避免只丢大段代码让人猜意图。
- **工业安全相关**（`Safety`、`StateMachine` 取消/超时语义、`Data` 迁移兼容）：必须交代**边界条件与风险**（例如 AABB 近似、SQLite 迁移限制），否则视为交付不完整。
- **不留假完成**：禁止 `TODO`、假数据、占位接口冒充可用实现；如信息不足，先列出缺失输入与可选方案，再请用户选择。
- **Halcon / UI 错误处理**：Halcon 异常需捕获并记录；若改动触及 UI，需可视化反馈（或由调用方捕获并提示），避免静默失败。

### 例外（仅在用户明确指出时）

- 若用户明确要求 **「只要代码不要解释」**：可在该会话内收敛说明，但仍需保证代码本身是完整可用的，且不得以省略关键分支的方式“看起来很短”。

## 禁止行为（默认）

- **文件系统**：不为了“省事”批量创建/删除/移动文件；仅允许完成当前任务所需的**最小**文件变更，且优先使用补丁/编辑器变更而非无关脚手架。
- **Git**：参见上文「提交规范」——默认不自作主张 `commit/push`，用户明确要求时再执行。
- **数据**：不编造设备参数、MES 返回值、Halcon 结果等；需要示例时用明显标记的示意值，并与真实生产配置隔离。
- **讨论范围**：不写与当前任务无关的长篇架构论文；但若涉及线程安全、同步 IO、序列化兼容性，必须用几句话说明关键点（这不是“泛泛架构”，是可维护性必要条件）。
